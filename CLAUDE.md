## Project

**Cog & Cosmos — The Fortune Engine** is a pixel-art interconnected incremental/idle game.
Eight stages (Village → Multiverse) each run an idle economy and feed each other; a central
**Fortune Engine** mints the universal currency **Fortune (★)** from every stage's surplus.

Built entirely with **free tools**. Current state: **Phase 3 (in progress)** — all eight live, interdependent
stages (**Village → Farm → Mine → Factory → Magic → Space → Time → Multiverse**), the **Fortune Engine**, a
**Global Skill Tree** (spend ★) + per-stage local upgrade trees, signature stage twists (Magic enchants, Space
input-chains, Time warp-tick bursts + Paradox, Multiverse duplication + Convergence collapse), prestige-gated
auto-buyers (cheapest + smart-priority modes), a Statistics panel + an **Achievements** system, the
**Ascension** meta-layer (deep reset → **Legacy Points** + an LP tree) and the **Transcendence** meta-layer
(deeper reset → **Aether Æ** + an Æ tree), a stacking toast notification bus, an installable **PWA**, an
**SPA** Stages/Skills/Ascension/Transcendence/Stats/Settings view switch, offline progress, autosave, and
synthesized SFX. All four prestige tiers are live: Prestige → Ascension (Legacy Points) → Transcendence
(Aether Æ) → **Reality Reset (Omega Ω)** (deepest reset → Omega; +10%/Ω to global production & ★ mint + an Ω tree).

## Tech stack

- **Svelte 5** (runes) — reactive UI
- **TypeScript** (strict) — everywhere
- **PixiJS v8** — the pixel sprite canvas only (render layer, not game logic)
- **break_eternity.js** — big-number math (`Decimal`)
- **idb-keyval + lz-string** — compressed IndexedDB saves
- **Vite** + **vite-plugin-pwa** — build/dev + installable, offline-capable PWA

## Architecture decisions (deliberate — don't undo)

- **SPA, not MPA.** The game is one persistent runtime (a continuous loop + live state + Pixi).
  Navigation is *in-app view switching* (`view` state in `GameLayout`: stages / skills / …), never a
  page reload — a reload would kill the loop and re-init everything.
- **Browser-first, shipped as a PWA.** Installable + offline via the service worker. No native shell;
  Tauri/Capacitor wrappers are a *post-launch* option only. App icons are generated by
  `scripts/generate-icons.mjs` (pure-Node PNG, no deps) — re-run it if the brand mark changes.
  The UI is **responsive**: a ≤720px mobile reflow (scrollable nav/dial strips, single-column deck) and a
  ≤480px compact mode (Pixi hidden) live as `@media` blocks in `GameLayout` + each panel; `--tap-min`/safe-area
  tokens are in `app.css`. Desktop layout is unchanged — every rule is inside a `max-width` query.

## Operating rules (non-negotiable)

These override default tooling behavior. Treat them as standing orders.

1. **No AI co-authorship in git.** Never add a `Co-Authored-By` trailer or any
   "generated by Claude / an agent" attribution to commits, PRs, tags, or pushes. Every commit is
   authored solely by the human owner. This explicitly overrides the harness default.

2. **Keep the paper trail current.** Any change that touches gameplay, systems, tooling, or
   structure updates the docs *in the same pass* — at minimum [CHANGELOG.md](./CHANGELOG.md)
   under `[Unreleased]`, plus [README.md](./README.md), [ROADMAP.md](./ROADMAP.md), or
   [MASTER_PLAN.md](./MASTER_PLAN.md) whenever scope or design shifts. Docs never lag the code.

3. **Free, but best-in-class.** Only free tools, libraries, and services — nothing paid, ever.
   Within that constraint, always reach for the *strongest* current option and modern best
   practice; "free" is never an excuse for a weak solution. Research current free alternatives
   before settling on one.

4. **Performance is a feature — the core must stay fast and cool.** No lag, no CPU/GPU thrash,
   no fan-spinning / overheating. Concretely:
   - Run the simulation on a **fixed-step tick decoupled from render** (currently 20 Hz with a
     catch-up cap), so a 60/120/144 Hz display does not multiply CPU work.
   - Per tick do **O(active generators)** work, not O(everything); skip locked/zero entities.
   - **Never allocate in hot loops** (reuse styles/objects; pool where it matters).
   - Keep the **Pixi layer decorative and throttled** — it redraws only on real change and
     naturally pauses when the tab is hidden (rAF). It must never gate the sim.
   - Avoid layout thrash and synchronous reads in the loop; profile before adding per-frame work.
   - Target: silky on a low-end laptop and a mid-range phone.

### Where the key systems live (in `stores/game.svelte.ts`)

- **Cross-stage bindings** — `STAGE_BINDINGS` (data-driven). Each rule reads a source quantity from
  one stage and returns a multiplier that **compounds** on the target. Today: Farm ← Village Labor (√);
  Mine ← Village Labor (√) × Farm Grain (log); Factory ← Farm Grain (log). (Magic, Space, Time, and
  Multiverse express their synergy through *broadcast* mechanics — enchants, input-chains, warp-ticks, and
  Shard duplication — not passive bindings; Multiverse's duplication multiplier is folded into `bindingMultFor`
  on the live path.) `bindingMultFor()` feeds `StageEconomy.tick()/.rates()`;
  `bindingInfos()` feeds the header badges. Add a stage's synergy = one entry here.
- **Stage unlock flow** — `checkUnlocks()` (called once per frame). Opens a stage at a lifetime-currency
  threshold, **seeds** a little starting currency, auto-assigns a Fortune Engine bay, and pushes a
  stacking toast via `pushToast()`. Village starts unlocked; Farm @ 5k lifetime Coins; Mine @ 5k lifetime Grain;
  Factory @ 5k lifetime Ore; Magic @ 1M lifetime Widgets; Space @ 5e5 lifetime Ore + sustained Power;
  Time @ 5M lifetime Stardust once any two stages have ascended; Multiverse @ 10M lifetime Chronons once
  all 7 prior stages are open and Space supplies Alloy.
- **Global Skill Tree** — `buySkill()` spends ★, then `recomputeUpgrades()` rebuilds `globalMult`
  (production) and `engine.engineMult` (mint) deterministically from levels — so it's always save-safe.
  The **Ascension meta tree** (`buyAscensionSkill()`, spends **Legacy Points**) folds into the *same*
  `recomputeUpgrades()` pass, so both trees are recomputed-safe and never drift.
- **Ascension layer** — `ascendStage()` deep-resets a stage (generators, currencies, *and* its prestige
  currency/count) and credits the global `legacyPoints` pool via `ascensionGain()` (harsher `^0.33`).
- **Transcendence layer** — `transcend()` is the deeper, global reset: it spends every stage + LP + upgrades
  to mint **Aether (Æ)** (which passive-mints ★ at +5%/Æ). Its 4-node Æ tree (`buyTranscendenceSkill()`)
  folds into the *same* `recomputeUpgrades()` pass as the ★ and LP trees. `transcendPreview()` drives the UI.
- **Achievements** — `checkAchievements()` (once per frame) evaluates the predicates in `data/achievements.ts`;
  newly-unlocked IDs land in `unlockedAchievements`, compound a small global output multiplier, and push a 🏆 toast.
- **Reality Reset layer** — `realityReset()` is the deepest reset (above Transcendence): it wipes the live game
  *and* the Aether pool + `transcendCount` to mint **Omega (Ω)** via `omegaGain()` (cube-root of `aetherLifetime`),
  keeping the Æ tree. Its 4-node Ω tree (`buyOmegaSkill()`) folds into `recomputeUpgrades()`, and `1 + 0.10·Ω`
  multiplies *both* `globalMult` and `engine.engineMult`. `omegaPreview()` drives the UI.
- **SPA views** — `view` state in `GameLayout` swaps Stages ↔ Skills ↔ Ascension ↔ Transcendence ↔ Omega ↔ Challenges ↔ Stats ↔ Settings, no reload.
- **Challenges** — `enterChallenge()` stashes the real save (`exportSave`) and swaps `gs` to a fresh restricted run;
  `maybeCompleteChallenge()` (called from `checkUnlocks`) restores it (`importSave`) and credits **Medals** on the
  real save. `activeChallengeRestriction()` gates auto-buy / bindings / `prodMult` / prestige at their hook points;
  the `ch:*` Medal tree (`buyChallengeSkill()`) folds into `recomputeUpgrades()` and survives every reset.
- **Auto-buyer** — per-stage `autoBuy` flag (unlocks after a stage's first prestige). `stepSim()` calls
  `economy.autoBuyTick()` once per step. Two modes (`autoBuyMode`: `'cheapest' | 'priority'`) — the smart
  ΔRate/cost **priority** mode unlocks at first Ascension, with a spend reserve and a per-generator vault.
  Toggle via `toggleAutoBuy()`.

## Conventions & hard rules

1. **Never use raw `number` for currencies or costs.** All game values are `Decimal`
   (break_eternity). Import `Decimal`, `D`, `ZERO`, `ONE`, `fmt` from `systems/Decimal.ts` — never
   import `break_eternity.js` directly elsewhere.

2. **All balancing math goes in `systems/formulas.ts`** as a pure function. Don't inline magic
   numbers in components or economies. The numbers must stay consistent with MASTER_PLAN §17.

3. **Svelte 5 module-rune gotchas** (these will break the build — learned the hard way):
   - You **cannot `export` a raw `$state` or `$derived`** from a `.svelte.ts` module.
     Export an **accessor function** instead: `export function village() { return gs.stages.village }`,
     then call it inside a component's `$derived(...)`.
   - `{@const}` in markup must be the immediate child of a block (`{#if}`, `{#each}`, etc.),
     never a direct child of a plain element. Lift it into the `<script>` as a `$derived` instead.

4. **Components are render-only.** They call store functions (`buyGenerator`, `prestigeStage`, …)
   and read accessor functions. No economy math in `.svelte` files.

5. **Pixi is decorative.** Never put authoritative game state in PixiJS. It reads the store each
   tick and draws; if it vanished, the game would still be correct.

6. **Styling** uses the `app.css` design tokens (CSS variables). Don't hardcode hex colors in
   components — use `var(--brass)`, `var(--ink-800)`, the per-stage `--sc` accent, etc.
   Fonts: `--font-display` (Silkscreen), `--font-mono` (Spline Sans Mono), `--font-flavor` (Newsreader).

7. **Saves are versioned (currently v13).** If you change the `GameState` shape, bump
   `CURRENT_VERSION` in `SaveManager.ts` and add a migration step in `migrate()`. `Decimal` values
   serialize via the `{ __decimal__: "..." }` wrapper — keep that working. Derived values
   (`globalMult`, `engineMult`) are **recomputed on load** from skills, not trusted from the save.
   Note: `RESET_BELOW_VERSION` lets `loadGame()` hard-reset pre-rebalance saves instead of migrating
   them (used for the v5 economy rebalance); raise it only for another deliberate clean-slate change.

8. **Derived multipliers are recomputed, not authoritative.** `globalMult` and `engine.engineMult`
   come from `recomputeUpgrades()` (skills). Don't set them by hand — change a skill/level and recompute.
   **Exception:** `convergenceMult` (the permanent ×mult baked by Multiverse's Convergence collapse) is
   *persisted, not derived* — it's only ever multiplied by a collapse, and `effGlobalMult()` folds it into
   the effective global multiplier at each production call-site. Don't move it into `recomputeUpgrades()`.

## How to add a new stage (extensibility recipe)

Use `farm.ts` / `mine.ts` as the templates — they already show every step.

1. **Create** `src/data/stages/<stage>.ts` exporting a `StageDefinition` (copy `mine.ts`).
   Use the canonical names/currencies/`fortuneWeight` from MASTER_PLAN §"Canonical Game Spec".
   Secondary-currency generators use `secondaryRate`; cross-stage exporters use `laborOutput`.
2. **Register** in `stores/game.svelte.ts`: add to `STAGE_DEFS`, `STAGE_ECONOMIES`, and the
   `stages: {}` of `freshGameState()`. (`ECONOMY_LIST` order = tick order — register a producer
   *before* the stage that reads its output, so bindings see fresh values.)
3. **Roster** — add the stage to `src/data/roster.ts` so its dial appears (locked until unlocked).
4. **Unlock** — add a clause to `checkUnlocks()`: gate on the prior stage's `primaryLifetime`,
   seed a little starting currency, and `fortuneEngine.assignSlot(...)`. Set `unlocked:false` in the
   stage's `freshState()` (only Village is `true`).
5. **Binding** — add an entry to `STAGE_BINDINGS` (define a `BindingRule` that reads a source stage
   and returns a multiplier). It auto-flows into ticking, rates, and the header badges.
6. **Pixi scene (optional)** — add an `is<Stage>` branch in `PixiCanvas.svelte`, or it falls back fine.
7. `npm run build` to verify.

## What NOT to do

- Don't add paid dependencies or services. Free tools only — this is a hard project constraint.
- Don't bypass `formulas.ts` with ad-hoc math.
- Don't store `Decimal` as `number` anywhere persistent — precision loss breaks the late game.
- Don't make Pixi authoritative or block the game loop on rendering.
- Don't claim "it works" without an actual `npm run build` (and ideally a dev-server check).
